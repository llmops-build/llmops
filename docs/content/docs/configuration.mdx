The `llmops()` function accepts a configuration object that controls how LLMOps behaves. This page documents all available options.

## Basic Example

```typescript title="llmops.ts"
import { llmops } from '@llmops/sdk';
import { Pool } from 'pg';

export default llmops({
  basePath: '/llmops',
  database: new Pool({
    connectionString: process.env.DATABASE_URL,
  }),
});
```

---

## Configuration Options

### basePath

The URL path where LLMOps routes will be mounted. Must start with `/`.

```typescript
basePath: '/llmops';
```

This determines where the dashboard and API endpoints are accessible:

- Dashboard: `http://localhost:3000/llmops`
- API: `http://localhost:3000/llmops/api/genai/v1/chat/completions`

---

### database

A PostgreSQL connection pool instance.

```typescript
import { Pool } from 'pg';

database: new Pool({
  connectionString: process.env.DATABASE_URL,
});
```

You can also pass individual connection options:

```typescript
database: new Pool({
  host: 'localhost',
  port: 5432,
  database: 'llmops',
  user: 'postgres',
  password: 'password',
});
```

---

### schema

The PostgreSQL schema name where LLMOps tables will be created. Defaults to `'llmops'`.

```typescript
schema: 'llmops'; // default
```

This option:

- Creates the schema if it doesn't exist (during migrations)
- Sets `search_path` on every database connection
- Isolates LLMOps tables from your application's tables

| Value      | Behavior                                  |
| ---------- | ----------------------------------------- |
| `'llmops'` | Uses dedicated `llmops` schema (default)  |
| `'public'` | Uses the default PostgreSQL public schema |
| `'custom'` | Uses a custom schema name of your choice  |

```typescript
// Use public schema (not recommended for production)
schema: 'public';

// Use a custom schema
schema: 'my_llmops_schema';
```

Using a dedicated schema keeps LLMOps tables separate from your application tables, making it easier to manage and backup.

## Provider Configuration

LLM providers are configured through the LLMOps dashboard, not in the configuration file. This allows you to:

- Add, update, or remove providers without code changes
- Securely store API keys in the database
- Change providers at runtime

### Setting Up Providers

1. Open the LLMOps dashboard
2. Navigate to **Settings** â†’ **Workspace Providers**
3. Click **Add Provider** and select your provider
4. Enter your API credentials and save

### Supported Providers

LLMOps supports 70+ AI providers including:

- **OpenAI** - GPT-4, GPT-3.5, etc.
- **Anthropic** - Claude 3, Claude 2, etc.
- **Google** - Gemini, Vertex AI
- **AWS Bedrock** - Claude, Llama, etc.
- **Azure OpenAI** - Azure-hosted OpenAI models
- **OpenRouter** - Access to 100+ models via single API
- **Groq** - Fast inference for open models
- **Together AI**, **Fireworks AI**, **DeepSeek**, and many more

See the [Providers](/docs/providers) page for the complete list and provider-specific configuration options.

## Complete Example

```typescript title="llmops.ts"
import 'dotenv/config';
import { llmops } from '@llmops/sdk';
import { Pool } from 'pg';

export default llmops({
  basePath: '/llmops',

  database: new Pool({
    connectionString: process.env.DATABASE_URL!,
  }),

  // PostgreSQL schema (defaults to 'llmops')
  schema: 'llmops',
});
```

## Environment Variables

We recommend storing sensitive values in environment variables:

```bash title=".env"
# Database
DATABASE_URL=postgresql://postgres:password@localhost:5432/llmops
```

Provider API keys are stored securely in the database after being configured through the dashboard.
